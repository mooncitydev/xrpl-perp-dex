# SGX Enclaves — возможности, ограничения и проблема миграции

**Статус:** Справочный документ / Q&A. Написан, чтобы зафиксировать, что команде нужно знать про SGX, прежде чем рассуждать о чём-то, что касается энклейва — в частности, прежде чем менять код, читающий или пишущий sealed state (например, vault).

**Охват:** практические основы SGX, релевантные этому проекту (perp DEX с FROST 2-of-3 подписью + sealed state). Не исчерпывающий SGX-справочник — для этого читайте Intel SGX Developer Reference и SDK-мануалы.

**Аудитория:** технические читатели (разработчики, операторы, аудиторы), которые раньше не работали с Trusted Execution Environment. Предварительных знаний про SGX не предполагается. Ожидается знакомство с концепциями ОС (процессы, syscalls, виртуальная память) и базовой криптографией (симметричное vs асимметричное, хеши, AES-GCM).

**Английская версия:** `sgx-enclave-capabilities-and-limits.md` (по правилу bilingual docs).

## Часть 0 — Рамка: зачем вообще Trusted Execution Environment?

### В: Я знаю, как работает обычный код. Какую проблему TEE решает, которую обычный код не может?

Подумайте, где живут приватные ключи в обычном сервере. Процесс-кошелёк запущен, держит ключ в RAM, подписывает транзакции. Любой, у кого **root на этой машине** — сисадмин, malware, скомпрометированный SSH, кто-то дампит `/proc/<pid>/mem` — может прочитать этот ключ. Обычные ОС никогда не были спроектированы, чтобы защищать процесс от собственного администратора. Root может читать любую память, какую хочет. Это не баг, это весь смысл ядра.

Для кода, держащего высокоценные секреты (signing-ключи, приватные данные пользователей, доступ к средствам), это проблема. Традиционный ответ — **физическая изоляция**: положить ключи на отдельный ящик (HSM, hardware wallet, air-gapped signer) и дать untrusted серверу просить trusted-ящик подписывать вещи по узкому проводу. Это работает, но дорого, медленно, и провод всё равно надо защищать.

**Trusted Execution Environment (TEE)** — другой ответ: положить чувствительный код в регион *той же* машины, но такой, в который даже ядро не может заглянуть. CPU сам обеспечивает изоляцию на уровне железа. Администратор видит, что процесс запущен, может его остановить, может удалить его файлы — но не может прочитать, что внутри него происходит во время исполнения. Получается что-то между "обычным процессом" и "отдельной физической машиной".

**Intel SGX** — одна конкретная реализация TEE. Есть и другие (AMD SEV, ARM TrustZone, Intel TDX), каждая со своими компромиссами. SGX работает на уровне *процесса*: обычный Linux-процесс может содержать небольшой изолированный регион, называемый **enclave**, и только код, загруженный в этот регион, работает в защищённом режиме.

### В: Дайте интуицию одной картинкой.

Обычный процесс — это дом. Операционная система — это город: она может сносить стены, читать вашу почту, видеть каждую комнату. TEE добавляет **комнату-сейф** внутри дома. Всё за пределами сейфа работает как обычно. Внутри сейфа никто — ни город, ни другой жилец, ни даже кто-то, кто вломился в дом с физическими инструментами — не может видеть, что происходит, пока вы готовы принять ограничения работы внутри сейфа (нет окон, нет телефонной линии наружу, сообщения передаются через щель в двери).

Аналогия сейфа несёт дальше, и мы будем использовать её многократно:

- Передача сообщений через щель → **ECALL** и **OCALL** (единственные способы перейти границу).
- "Нет телефонной линии наружу" → энклейв **не может делать syscalls**, включая сетевые операции.
- Купчая на сейф у конкретного человека → **enclave measurement (MRENCLAVE)** и **signing key measurement (MRSIGNER)**.
- Хранение документа так, чтобы только *вы* могли его потом прочитать → **sealing**.
- Доказать кому-то снаружи, что сейф существует и содержит то, что вы заявляете → **attestation**.

Если вы запомните сейф, всё остальное в этом документе ляжет.

### В: Почему именно perp DEX будет использовать это?

Потому что это позволяет signing-ключу, ledger'у маржи и политике риска жить в коде, в который оператор не может ни заглянуть, ни тихо его изменить. Оператор может стартовать и останавливать сервис, может читать зашифрованные blob'ы на диске, может видеть сетевой трафик — но собственно приватные ключи и собственно decision-making код закрыты от него. В FROST 2-of-3 эта property держится через трёх независимых операторов, каждый со своим энклейвом на своей машине. Ни один оператор в одиночку не может эксфильтровать ключи или подделать подпись, потому что ключей в виде, который оператор может прочитать, просто не существует.

Это главная причина, почему проект построен на SGX: **trust story в том, что никому из операторов — даже тому, кто запускает бинарь — не нужно доверять для безопасности протокола**.

### В: В чём подвох?

Несколько подвохов. Остальная часть документа проходится по ним. Кратко:

1. Enclaves не могут делать I/O. Это вычислительный ящик, а не сетевой.
2. Изменение кода энклейва ломает его identity, что ломает способность читать его собственные сохранённые данные.
3. У SGX были реальные уязвимости — он сильный, но не неломаемый.
4. Вы доверяете Intel (железу и attestation-инфраструктуре), хотите вы этого или нет.
5. Операционно SGX капризен. Microcode-апдейты, замены CPU, attestation-водопровод — у всего острые края.

С этой рамкой остальное — детали.

## Часть 1 — Что такое SGX на самом деле

### В: В одном абзаце — что такое SGX-энклейв?

SGX-энклейв — это **изолированная execution-область** внутри в остальном обычного Linux-процесса, защищённая самим CPU. Код и данные энклейва живут в специальном куске RAM, называемом **Enclave Page Cache (EPC)**. Железо автоматически шифрует каждый байт, записываемый в EPC, и дешифрует его только когда CPU исполняет код, принадлежащий этому конкретному энклейву; ключ шифрования генерируется внутри CPU при boot'е и никогда не покидает чип. Следствие: ОС, гипервизор, другие процессы, другие энклейвы и даже атакующий, физически тапающий memory bus пробой, **не могут прочитать память энклейва в plaintext**. Код энклейва хешируется во время загрузки (этот хеш называется **measurement**), и этот measurement криптографически привязан ко всему, что энклейв делает потом — это якорь, делающий возможным и "сохранить данные так, чтобы только этот код мог их прочитать" (sealing), и "доказать удалённой стороне, какой код запущен" (attestation).

### В: Какие ключевые security-свойства я реально получаю?

1. **Конфиденциальность** — память энклейва непрозрачна для всего, что вне него, включая ядро ОС.
2. **Целостность** — код энклейва нельзя молча подменить; любое изменение даёт другой measurement.
3. **Attestable identity** — энклейв может доказать удалённой стороне, что конкретный measurement кода работает в подлинном Intel SGX CPU.
4. **Связанная persistence** — данные, запечатанные энклейвом, могут быть распечатаны только энклейвом с соответствующей identity на том же семействе CPU.

### В: Чего SGX *не* даёт.

- **Это не контейнер и не VM.** Это изолированный compute-регион и ничего больше. Нет файловой системы, нет сетевых сокетов, нет возможности вызывать ядро. Единственное, что энклейв может делать — это то, что вы явно wire'ите как entry и exit points при компиляции.
- **Он не tamper-proof против ваших собственных багов.** Если код внутри энклейва имеет memory-safety баг или логическую ошибку, которая leak'ает секреты, SGX вас не спасёт. Гарантия конфиденциальности работает только если код самого энклейва случайно не публикует свои секреты. Именно поэтому enclave-код почти всегда маленький, аккуратный и тщательно review'ится.
- **Он не бесплатный в performance.** Каждый раз, когда обычный код входит в энклейв, CPU должен переключить режим, flush'нуть некоторое state и валидировать переход — несколько микросекунд на пересечение. Если плохо спроектируете — можете тратить больше времени на пересечение границы, чем на полезную работу внутри. EPC тоже конечен (см. Часть 8), и внутри энклейва нельзя использовать полную стандартную библиотеку.
- **У него есть product roadmap, как у любой платформы.** Intel депрекейтнул SGX на потребительских Core CPU 11/12-го поколения в 2021, но **server SGX на процессорах Xeon Scalable — first-class, активно поддерживаемая платформа**. Официальная позиция Intel (support-статья 000089326): *"no plans to deprecate Intel SGX on the supported Intel Xeon Scalable processors"*, разработка под cloud и enterprise продолжается — именно на это мы целимся. Новая **TDX** (Trust Domain Extensions) Intel — связанная технология, защищающая целые VM вместо отдельных процессов; она позиционируется как *дополнение*, а не замена server SGX, и это отдельная тема.
- **Он не магический ящик, который нельзя взломать — но его взлом это лабораторное упражнение, а не удалённый хак.** SGX радикально поднимает планку для атакующего, но реальные пробои были. Семейство атак под названием **side-channels** — где атакующий не читает память напрямую, а *выводит* её содержимое из тайминга, поведения кэша, электромагнитных наводок или потребления энергии — многократно работало против SGX. Для threat modelling важно *как* эти атаки реально выполняются на практике:
  - Практически каждая опубликованная в академической литературе успешная side-channel атака на SGX (Plundervolt, VoltPillager, Foreshadow, SGAxe, ÆPIC Leak, семейство power analysis) требует либо **физического обладания машиной** с подключённой инструментовкой (осциллографы, power-пробы, hardware для манипуляции напряжением, FPGA-интерпозеры на шину памяти), либо **привилегированного кода с root-правами на том же хосте** с часовой-суточной тонкой настройкой.
  - Ни одна из них не является "кликнул ссылку и вскрыл энклейв" удалённым эксплойтом. Чтобы провести такую атаку, атакующему нужно **физически увезти наш сервер к себе в лабораторию, подключить измерительную аппаратуру и работать над ним часами или сутками** — или сначала полностью скомпрометировать host OS и затем запустить тщательно настроенную локальную атаку. Оба сценария в scope нашей threat model (мы исходим из того, что host OS враждебна — в этом и смысл использования SGX), но они категорически отличаются от "кто-то в интернете забрал наши ключи", о чём беспокоится большинство web-систем.
  - Каждую опубликованную атаку Intel митигировал microcode-апдейтом, и каждая митигация продвигает счётчик **CPUSVN** (подробнее ниже) — то есть пропатченный сервер на актуальном microcode защищён от *известных* семейств атак.
  - Наш defence-in-depth ответ на остаточный риск — **FROST 2-of-3**: даже полный компромисс одного энклейва на одном сервере не раскрывает подписной ключ, потому что ни один энклейв целиком ключ никогда не держит.
  - Правильный способ думать о SGX: "драматически лучше обычного процесса, заставляет серьёзного атакующего либо ехать в лабораторию, либо становиться insider'ом, но всё равно требует того же defence-in-depth мышления, что и любая security-граница".

## Часть 2 — Архитектура и memory model

### В: Где энклейв реально живёт?

Внутри обычного Linux-процесса. Один процесс может содержать ноль, один или несколько энклейвов — это как специально защищённые "комнаты" внутри адресного пространства вашего процесса. Физическая память, за которой стоит энклейв, вырезана из региона, который BIOS резервирует при boot'е и который называется EPC (Enclave Page Cache). Каждый байт, который CPU записывает в EPC, автоматически шифруется на входе и дешифруется на выходе из CPU — выделенным hardware-блоком. Так что даже если бы вы заморозили RAM и сдампили его логическим анализатором, вы бы получили шифртекст, не plaintext. Ключ шифрования регенерируется внутри CPU при каждом boot'е и никогда не покидает чип.

### В: Как non-enclave код в том же процессе разговаривает с enclave-кодом?

Через два типа контролируемых переходов, определяемых во время компиляции в маленьком файле под названием **EDL** (Enclave Definition Language). Думайте об EDL как о декларации "щелей в двери сейфа": каждая функция, которую внешнему миру разрешено просить энклейв выполнить, и каждая функция, которую энклейв может вызвать обратно в внешний мир, чтобы попросить сделать что-то от своего имени.

- **ECALL** — untrusted host → trusted enclave. "Host, пожалуйста, попроси энклейв сделать X." Host передаёт аргументы и получает return value, похоже на обычный вызов функции, но это на самом деле контролируемый вход в защищённый регион.
- **OCALL** — trusted enclave → untrusted host. "Enclave, попроси host сделать Y от моего имени и верни результат." Используется для всего, что энклейв не может делать напрямую — а это, как мы сейчас увидим, "почти всё, что касается внешнего мира".

Каждый ECALL или OCALL — это **context switch**: CPU должен flush'нуть некоторое pipeline state, сменить режим выполнения и re-валидировать переход. Это существенно дороже обычного вызова функции, обычно несколько микросекунд. Медленно per-call, нормально per-transaction. Хорошо спроектированный энклейв вызывают редко с "толстыми" аргументами, а не часто с "тонкими".

### В: Может ли энклейв напрямую делать syscalls?

**Нет, и это by design.** **Syscall** — это как обычный user-space код просит ядро делать привилегированные вещи: читать файл, слать сетевой пакет. Нормально вы вызываете `read()`, CPU переходит в kernel mode, ядро делает работу, вы получаете байты обратно. Если бы энклейв мог это делать, ядро — которое явно вне trust boundary энклейва — увидело бы аргументы запроса, контролировало бы ответ и могло бы легко врать или подсматривать. Поэтому Intel заблокировал это: CPU буквально отказывается исполнять инструкции `syscall` изнутри энклейва.

Следствие: каждая syscall-образная операция — чтение файла, открытие сокета, запрос текущего времени — должна происходить *снаружи* энклейва, и энклейв должен OCALL'ить и попросить host сделать это. Host untrusted, так что энклейв должен относиться к каждому OCALL-ответу как к потенциально вредоносному и валидировать его.

Это самое важное ограничение для рассуждений о том, что принадлежит внутри энклейва, а что снаружи. Если ваша фича должна говорить с сетью — она живёт в host. Если ваша фича должна держать секрет и принимать решения о нём — она живёт в энклейве. Всё промежуточное — это design choice о том, где провести границу.

## Часть 3 — Сетевое ограничение (и почему энклейв не может сам слать XRPL-транзакции)

### В: Значит, энклейв реально не может открыть сокет?

Верно. Внутри энклейва нет TCP/UDP-стека, нет DNS-резолвера, нет TLS-библиотеки, которая говорит с чем-то кроме как через OCALL'енные byte streams. Если вам кто-то говорит "энклейв подключается к XRPL" — он использует слово вольно: на самом деле имеется в виду "энклейв производит подписанные байты, *host* шлёт их в XRPL и получает байты обратно, которые relay'ит в энклейв через ECALL".

### В: Почему это важно для signer'а?

Потому что это значит, что signer **не может автономно постить транзакции**. Flow для любой подписанной операции всегда такой:

1. Host (Python, Rust, whatever) строит template транзакции.
2. Host делает ECALL в энклейв: "подпиши эту tx ключом пользователя U, соблюдая политику P".
3. Энклейв валидирует политику, читает свой sealed key material, подписывает, возвращает подписанные байты через return value ECALL'а.
4. Host берёт подписанные байты и submit'ит их в XRPL через любой RPC/WebSocket клиент, который использует.
5. Host ждёт XRPL-подтверждения, вытаскивает результат, и снова ECALL'ит энклейв, если энклейву нужно знать о результате ("tx hash X подтверждён в ledger L").

Энклейв никогда не видит wire. Энклейв никогда не инициирует. Энклейв только **реагирует** на ECALL'ы и **делегирует** сетевые операции через OCALL'ы.

### В: Значит ли это, что host может withhold транзакции, которые энклейв подписал?

Да, и это важное честное ограничение. Если host скомпрометирован или вредоносен, он может:
- Sign-and-discard (попросить энклейв подписать, потом выбросить подписанные байты).
- Sign-and-delay (держать подпись, пока рынок не двинется).
- Sign-and-replay (пересабмитить старую подпись, если nonce/sequence позволяет).

Энклейв не может предотвратить это напрямую, потому что не видит сеть. Что он *может* — сделать эти атаки детектируемыми или бесполезными:

- **Sequence / nonce дисциплина** — если каждая подписанная tx должна ссылаться на монотонно растущий счётчик, который трекает энклейв, replay блокируется на уровне энклейва.
- **Deadlines, вшитые в подписанные blob'ы** — включать `LastLedgerSequence` (native XRPL) или эквивалент, чтобы устаревшие подписи истекали.
- **Cross-operator attestation** — в FROST 2-of-3 нужны двое из трёх операторов для подписи, так что один malicious host не может сфабриковать подпись в принципе.
- **Audit trail** — энклейв может логировать tx hashes подписанных транзакций в своё собственное sealed state, чтобы аудитор потом проверил, что каждая sealed-логированная tx появляется on-chain.

Именно поэтому наша custody — FROST, а не single-enclave signer: атака "скомпрометированный host withhold'ит подписанные байты" митигируется требованием двух независимых host'ов сотрудничать.

### В: А время? Может ли энклейв доверять своим часам?

**Нет.** Внутри SGX-энклейва нет надёжных часов. CPU не экспонирует монотонный secure counter, который был бы свободно читаем. В SGX есть "trusted time" сервис (`sgx_get_trusted_time`), но он депрекейтнут и никогда не был надёжным. На практике время приходит от untrusted host через OCALL, что означает, что вредоносный host может врать о том, какое сейчас время. Защита: для всего time-sensitive энклейв должен требовать *подписанный* timestamp от внешнего источника (XRPL ledger close time — приличный выбор, так как он привязан к ledger'у, который host не может подделать без сговора со всей сетью).

## Часть 4 — Sealing: как данные шифруются и расшифровываются

### В: Если у энклейва нет файловой системы, как он что-то persist'ит?

На первый взгляд это выглядит невозможным. Энклейв не может вызвать `write()`, так что не может ничего сохранить на диск. Но он может OCALL'ить в host и попросить host сохранить opaque blob байтов.

Фокус в том, что прежде чем передать байты host'у, энклейв **шифрует их внутри себя ключом, который может derive только энклейв с правильной identity**. Host получает шифртекст, хранит его как хочет, и никак не может его прочитать. Позже этот же энклейв (или другой, который CPU считает эквивалентным) просит у host'а шифртекст обратно, re-derive'ит тот же ключ и расшифровывает.

Весь этот танец — шифрование внутри, хранение снаружи, расшифровка снова внутри — называется **sealing**. Думайте об этом как о том, что энклейв кладёт что-то в запертый ящик, который только hardware-derived ключ этого конкретного энклейва может открыть, и потом позволяет untrusted-миру таскать этот запертый ящик как хранилище.

Магия целиком в том, *откуда берётся ключ*. Ключ нигде не хранится. Он derive'ится каждый раз из:
1. Master secret, который Intel прожёг в CPU на фабрике.
2. Identity энклейва, запрашивающего ключ.
3. Нескольких version numbers.

Ни один энклейв на любом другом CPU не может derive тот же ключ, потому что master secret другой. Ни один другой энклейв на том же CPU не может derive тот же ключ, потому что enclave identity другая. Именно поэтому sealed-данные привязаны к **одному конкретному enclave binary на одном конкретном физическом CPU**.

### В: Проведи меня через seal-операцию end-to-end.

1. У энклейва есть какие-то байты, которые он хочет сохранить — например, приватный key share или sealed vault state blob.
2. Энклейв вызывает `sgx_seal_data()` (или эквивалент в том SGX SDK, который вы используете).
3. CPU derive'ит **sealing key** из секрета, зашитого в CPU во время производства (root sealing key), в комбинации с measurement энклейва (об этом через момент) и CPU security-version info (CPUSVN, ISV_SVN).
4. Энклейв использует этот derived key с AES-GCM (или AES-CCM), чтобы произвести ciphertext blob с integrity tag.
5. Энклейв OCALL'ит: "host, пожалуйста, запиши этот blob в файл `/path/to/sealed.bin`".
6. Host пишет blob. Host не может прочитать plaintext — у него никогда не было ключа, и ключ никогда не был вне CPU.

### В: А unsealing?

Обратное:

1. Host читает blob с диска, ECALL'ит: "вот sealed blob".
2. Энклейв вызывает `sgx_unseal_data()`.
3. CPU re-derive'ит sealing key из тех же inputs (root key + measurement + CPUSVN).
4. Если derived key совпадает — AES-GCM расшифровывает и integrity-verify'ит, и энклейв получает plaintext.
5. Если какой-то из inputs отличается — другой enclave binary, другой CPU, другой CPUSVN — key derivation производит другой ключ, и дешифровка фейлится.

Последнее предложение — это вся проблема миграции в одной строке. Держите её в уме.

### В: В чём разница между MRENCLAVE-sealing и MRSIGNER-sealing?

Sealing-key derivation может быть привязан к одной из двух enclave identities:

- **MRENCLAVE** — SHA-256 measurement кода энклейва и его initial data. Измените один байт своего enclave binary, и MRENCLAVE меняется. Sealing на MRENCLAVE означает: *только этот точный binary, пересобранный бит-в-бит идентично, может когда-либо распечатать этот blob*.

- **MRSIGNER** — SHA-256 хеш Intel-формат публичного ключа, использованного для подписи энклейва. Каждый энклейв, подписанный тем же ключом, имеет тот же MRSIGNER, вне зависимости от различий в коде. Sealing на MRSIGNER означает: *любой энклейв, подписанный нами, включая будущие версии, может распечатать этот blob*.

SDK экспонирует этот выбор как параметр `policy` для seal-вызова.

### В: В чём trade-off?

| Свойство | MRENCLAVE sealing | MRSIGNER sealing |
|---|---|---|
| Tamper resistance | Очень высокая — любое изменение кода инвалидирует данные | Ниже — любой подписанный нами энклейв может их прочитать, включая багованную новую версию |
| Upgrade path | **Нельзя апгрейдиться без явной миграции** | Апгрейды "просто работают", пока signing key не меняется |
| Худший случай при утечке signing key | Данные в безопасности — атакующему нужен точный binary | Атакующий может собрать malicious энклейв, подписанный leaked ключом, и прочитать всё |
| Типичное применение | Ephemeral секреты, test keys, per-version state | Долгоживущие ключи, user databases, всё, что должно пережить апгрейды |

На практике долгоживущий persistent state (как база позиций vault'а) использует **MRSIGNER** sealing. Short-lived или version-specific state использует **MRENCLAVE** sealing. Выбор делается per-blob, не глобально.

### В: Что ещё влияет на sealing key?

Да, и вот где людей кусает:

- **CPUSVN** — security version number CPU, который продвигается, когда microcode добавляет митигации для новых side-channel атак. Key derivation использует *текущий* CPUSVN, а не тот, что был на момент sealing'а. Есть режим "rollback resistance", где можно derive key для более старого CPUSVN (так что данные, запечатанные под CPUSVN=5, всё ещё читаемы после bump'а до CPUSVN=6), и строгий режим, где нельзя. Строгий режим — для защиты от microcode downgrade'ов; rollback-режим — для выживаемости через microcode-апдейты.

- **ISV_SVN** — application-level security version number, который вы встраиваете в metadata энклейва. Как CPUSVN, но под вашим контролем. Полезно для "patches 1–3 могут читать данные друг друга, но версия 4 не может быть прочитана версиями 1–3".

- **Сам CPU.** Sealing keys per-CPU. Переместите диск на другую машину — и sealing фейлится, даже если вы пересобрали точно тот же энклейв. Это намеренно — предотвращает клонирование диска. Это также означает, что **нельзя забэкапить sealed blob'ы, скопировав их на другой сервер, и ожидать, что они там заработают**. Backup требует явного export-механизма (подробнее в Части 7).

### В: Какой же lifecycle "сохранить позиции vault'а"?

Предполагая MRSIGNER sealing и предполагая, что мы ещё не пересобрали энклейв:

1. У энклейва есть vault state в RAM.
2. ECALL `save_state` срабатывает.
3. Энклейв сериализует state в байты.
4. Энклейв вызывает seal с MRSIGNER policy → ciphertext blob.
5. Энклейв OCALL'ит `write_file(blob)`.
6. Host пишет.

Позже:

7. Host читает файл.
8. ECALL `load_state(blob)`.
9. Энклейв вызывает unseal — успех, потому что тот же CPU + тот же MRSIGNER + совместимый CPUSVN.
10. Энклейв получает vault state обратно в RAM.

Это happy path. Всё, что следует в Части 7, о том, что происходит, когда один из inputs меняется.

## Часть 5 — Attestation: доказать удалённой стороне, что запущено

### В: Краткая сводка — какую проблему решает attestation и что это?

**Проблема:** представьте, что вы пользователь и хотите внести средства в наш протокол. Открываете соединение с `api-perp.ph18.io`. Отвечает сервер. Откуда вы знаете, что код, отвечающий вам, — это действительно энклейв, который мы заявляем, а не вредоносный оркестратор, выдающий себя за него? Нельзя просто посмотреть на код — он работает на чьей-то машине. Вам нужно, чтобы сама машина поручилась за то, что она запускает, в виде, который вы криптографически можете проверить.

**Решение:** **attestation** — механизм, при помощи которого энклейв, с помощью CPU, на котором он запущен, производит маленький подписанный документ (называемый **quote**), который говорит:

- "Конкретный кусок кода (идентифицированный по measurement) запущен."
- "Он работает внутри подлинного Intel SGX CPU, а не в эмуляторе или подделке."
- "Энклейв прикрепил вот эти 64 байта произвольных данных к доказательству" (это называется **report data**, и обычно это хеш свежесгенерированного публичного ключа, чтобы пользователь потом мог установить зашифрованный канал к энклейву).

Третья сторона, если доверяет Intel как корню цепочки, может верифицировать подпись quote'а и потом верить, что публичный ключ в report data действительно пришёл от измеренного энклейва на реальном SGX CPU. С этого момента они могут шифровать сообщения в энклейв, которые только энклейв может прочитать.

Коротко: attestation — это как энклейв представляется по untrusted-сети так, чтобы его нельзя было выдать за кого-то другого.

### В: Local vs remote attestation?

- **Local attestation** — энклейв A доказывает энклейву B на том же CPU, что A — это measurement, который он claim'ит. Быстро, Intel-сервисы не задействованы. Используется для intra-machine enclave-to-enclave обмена ключами.
- **Remote attestation** — энклейв доказывает стороне на другой машине. Есть два поколения: **EPID** (более старое, Intel'овская privacy-preserving group signature схема, нуждается в Intel attestation service) и **DCAP** (более новое, стандартное PKI-based, может быть верифицировано offline после bootstrap, используется современным Azure Confidential Computing и нашим setup'ом).

### В: Что входит в DCAP quote?

Упрощённо: quote — это структура, подписанная per-CPU attestation ключом (provisioned Intel'ом), содержащая MRENCLAVE, MRSIGNER, CPUSVN, ISV_SVN, набор флагов и 64 байта report data. Для верификации вы цепляете подпись к Intel root сертификату и потом применяете свою политику (принимаем ли мы этот MRENCLAVE? этот MRSIGNER? требуем ли CPUSVN ≥ N?).

### В: Для чего мы используем attestation в этом проекте?

Две вещи:

1. **DCAP attestation endpoint** — orchestrator экспонирует REST-эндпоинт, который возвращает свежий quote, доказывающий, что энклейв запущен. Клиент может верифицировать это до того, как доверить оркестратору депозиты.
2. **Cross-node attestation для FROST** — два энклейва, собирающиеся выполнять distributed signing, сначала верифицируют quote'ы друг друга. Если пир поднимается с неожиданным MRENCLAVE (скажем, потому что кто-то развернул backdoor'енный build на этот узел) — другие пиры отказываются с ним подписывать. См. `deployment-procedure.md` раздел 3.5 о том, как это интегрируется с deploy pipeline.

## Часть 6 — Проблема миграции версий (самое сложное)

Это та секция, которую пользователь специально попросил. Это то, что кусает каждый SGX-проект рано или поздно. Если вы прочитаете только одну часть этого документа — читайте эту.

### В: Переформулируй проблему в одном предложении.

Если вы меняете код внутри энклейва — даже один байт — MRENCLAVE меняется, и любой state, который был запечатан под MRENCLAVE, больше не может быть распечатан новым энклейвом. Даже если вы используете MRSIGNER sealing, баги в schema compatibility, выборе ISV_SVN или CPUSVN bumps всё равно могут запереть ваши старые данные.

### В: Что именно меняется, когда я редактирую vault code?

Допустим, вы редактируете `PerpEngine.cpp`, чтобы изменить расчёт liquidation threshold. Пересобираете энклейв. Новый `.so` файл имеет:

- Новый MRENCLAVE (потому что байты кода изменились).
- Тот же MRSIGNER (потому что вы использовали тот же signing key).
- Возможно новый ISV_SVN, если вы bump'нули его в metadata.

Что теперь происходит с sealed state на диске (скажем, файлом позиций vault'а)?

- Если позиции были **sealed под MRENCLAVE**: новый энклейв **вообще не может их прочитать**. Unseal фейлится. У вас brick.
- Если позиции были **sealed под MRSIGNER** с совместимым ISV_SVN: новый энклейв *может* распечатать blob. Но:
  - **Schema** внутри plaintext может не совпасть с тем, что ожидает новый код. Если вы добавили поле в `PerpPosition`, serialiser внутри нового энклейва прочитает старый blob как malformed версию-N-1 структуры, тихо искажая данные. Это отдельная проблема от sealing'а, и обычно именно она вас убивает на практике.
- Если вы также bump'нули CPUSVN (обновив microcode): в зависимости от seal policy вы тоже можете потерять доступ.

Так что проблема миграции имеет **три слоя**, сложенные друг на друга:

1. **Sealing key derivation должен всё ещё успешно проходить** → MRSIGNER policy + CPUSVN rollback tolerance.
2. **Schema должна быть versioned и backwards-readable** → каждая persisted структура несёт version byte, и новый энклейв имеет явный код для чтения старых версий и апгрейда их в памяти.
3. **Invariant changes должны быть явно мигрированы** → если ваш новый код предполагает "у каждой позиции есть поле `funding_index`", а у старых позиций его нет — нужен migration step, который вычисляет или default'ит поле при загрузке.

Пропустите любой из трёх — и потеряли данные.

### В: Какой стандартный совет люди получают и почему он недостаточен?

Наивный совет — "используйте MRSIGNER sealing и version'ьте ваши schemas". Это покрывает слои 1 и 2, но игнорирует слой 3 (invariant migrations), а также игнорирует более тонкую проблему, что MRSIGNER sealing ослабляет вашу tamper-resistance историю — тот, кто компрометирует signing key, может прочитать все исторические данные.

Для нашего проекта конкретно: signing key для энклейва живёт у нас (77ph personal account для enclave repo, и нормальный HSM-backed key для продакшена). Безопасность этого ключа становится полом безопасности каждого sealed blob'а, на всё время жизни проекта.

### В: Проведи меня через конкретную миграцию vault'а — v1.4.2 vault'а хочет добавить новое поле `last_funding_ts` к каждой открытой позиции. Как мы реально мигрируем?

Предположим, мы используем MRSIGNER sealing с versioned on-disk форматом. On-disk blob vault'а выглядит так:

```
[4-byte magic "VALT"]
[2-byte version, например, 0x0001]
[payload bytes, формат зависит от version]
[AES-GCM tag — добавлен SDK seal-операцией]
```

Когда новый энклейв грузится, он делает это:

1. OCALL: читаем blob с диска.
2. `sgx_unseal_data()` с MRSIGNER policy.
   - Если это фейлится, потому что CPUSVN продвинулся вперёд → логируем, bail out, требуем ручного восстановления (секция дальше ниже).
   - Если успех → у нас есть plaintext байты.
3. Парсим magic, проверяем, что совпадает с `"VALT"`. Если нет — отказываемся грузиться, это либо corruption, либо blob из другого энклейва.
4. Парсим поле version.
5. Если `version == 0x0002` (текущая) → десериализуем напрямую.
6. Если `version == 0x0001` (предыдущая) → вызываем **v1→v2 migrator**. Мигратор:
   a. Читает старый layout (без поля `last_funding_ts`).
   b. Конструирует новый layout в памяти.
   c. Заполняет `last_funding_ts` разумным default'ом — вероятно ledger time, в который стартовал новый binary, или ноль с явной семантикой "treat as never-updated".
   d. Возвращает новую структуру.
7. Новый энклейв нормально оперирует на апгрейдутом in-memory state.
8. Первый раз, когда новый энклейв вызывает `save_state`, он пишет version `0x0002`. С этого момента disk-формат навсегда апгрейднут.

**Rollback асимметричен.** Если вам когда-нибудь понадобится откатиться к v1.4.1 — эта версия никогда не видела новый формат и не может читать blob'ы версии `0x0002`. Либо:
- Сохраняйте старый blob как backup до первого upgrade save (обязательный операционный шаг), либо
- Пишите **downgrader** в v1.4.2, который может re-сериализовать в v1 формат по запросу, для rollback-emergency.

Downgrader редко стоит инженерных затрат, если только schema change не настолько опасен, что вы хотите быстрый rollback path.

### В: Что если новый энклейв должен поменять саму sealing policy (скажем, перейти с MRENCLAVE на MRSIGNER)?

Это паттерн **dual-mode transition**:

- Фаза 0 — всё sealed с MRENCLAVE. Вы понимаете, что вам нужны upgrades, и решаете переключиться на MRSIGNER.
- Фаза 1 — выкатываете **transition enclave** (назовём v1.9), который при старте:
  1. Читает старый MRENCLAVE-sealed blob.
  2. Распечатывает его используя MRENCLAVE policy.
  3. Re-seal'ит тот же plaintext, используя MRSIGNER policy.
  4. Пишет re-sealed blob под новым именем файла (`state.mrsigner.sealed`), оставляя старый файл нетронутым как safety net.
  5. С этого момента пишет только MRSIGNER-sealed blob'ы.
- Фаза 2 — v2.0 и более поздние энклейвы знают только про MRSIGNER sealing. Старый MRENCLAVE blob удаляется после того, как прошло достаточно операционного времени, чтобы быть уверенным, что новый формат работает.

Критическое свойство: **transition enclave запускается ровно один раз на каждом хосте для выполнения re-seal'а**. Критически, transition enclave сам должен быть собран из того же коммита исходников, что и v1 энклейва (чтобы его MRENCLAVE совпадал с тем, что sealed старый blob), и должен также знать v2 sealing policy (чтобы мог re-seal вперёд). Именно поэтому version transitions иногда имплементируют как *церемонию* — вы собираете и запускаете одноразовый binary только для перехода, потом выбрасываете его.

### В: Что если я полностью пересобираю signing key энклейва? (Скажем, старый signing key был скомпрометирован и мы ротируем на новый.)

Это ядерный случай. Новый signing key → новый MRSIGNER → **ни один из ваших существующих MRSIGNER-sealed blob'ов не может быть прочитан новым энклейвом, никогда**.

Единственный митигирующий путь — экспортировать plaintext через старый энклейв до ротации:

1. Соберите **ceremony энклейв для ротации ключа** используя старый signing key. У этого энклейва одна работа: распечатать всё и re-export plaintext в форме, которую новый энклейв может потребить.
2. Ceremony энклейв должен re-export plaintext *без* экспозиции его untrusted host'у — это ломает всю суть. Стандартный способ сделать это:
   a. Старый ceremony энклейв и новый энклейв оба запущены на одной машине одновременно.
   b. Они выполняют **local attestation**, чтобы верифицировать identity друг друга.
   c. Они устанавливают authenticated ephemeral key exchange (ECDH внутри local attestation).
   d. Старый энклейв распечатывает, шлёт plaintext через attested канал в новый энклейв.
   e. Новый энклейв re-seal'ит под новым MRSIGNER.
3. После церемонии старые sealed файлы удаляются.

Для FROST 2-of-3 setup эта церемония должна быть выполнена на каждом из трёх узлов, идеально одновременно или хотя бы до возобновления signing quorum, чтобы кластер не застрял с одним узлом на old-key-sealed state и двумя на new-key-sealed state.

### В: Что если я просто YOLO пересобираю и новый энклейв не может прочитать старый blob?

Зависит от того, что было в blob'е:

- **FROST key shares** — катастрофично. Вы больше не можете ничего подписывать. Вы также не можете участвовать в FROST quorum, потому что у вас нет share. Единственное восстановление — DKG с нуля, что означает новый аккаунт (новый multisig-адрес), что означает, что любые средства в старом аккаунте заперты, если только threshold выживших операторов не может всё ещё подписать recovery tx на старом аккаунте (что требует, чтобы *их* энклейвы всё ещё работали, т.е. они не YOLO'или одновременно).
- **Vault state** — плохо, но не терминально. Вы можете реконструировать position state из истории транзакций XRPL (каждая позиция была результатом подписанной tx on chain) со значительными ручными усилиями. Вы теряете любой in-enclave state, который не был on-chain (например, неотправленные ордера, volatile internal counters).
- **User margin ledger** — очень плохо. В отличие от vault, margin мутируется внутренней логикой между on-chain событиями, так что нельзя реконструировать его чисто из chain history, не replay'я каждый депозит, вывод, funding payment и fee по порядку. Возможно, но хрупко.

**Операционное правило:** прежде чем менять что-либо внутри энклейва, понимайте, в какую категорию попадает каждый sealed blob, и имейте план миграции для каждого из них *до* bump'а версии.

### В: Есть ли вообще способ сделать sealed data портируемой между CPU?

Не напрямую. Sealing key derive'ится из per-CPU root. Если хотите, чтобы sealed blob пережил CPU swap (или чтобы реплицировать энклейв на второй хост), придётся построить **export/import** pathway:

- Export: энклейв на CPU A распечатывает свои данные, шифрует их ключом, предоставленным через remote-attested канал от энклейва на CPU B, и host пайп'ит ciphertext с A на B.
- Import: энклейв на CPU B расшифровывает, верифицирует, re-seal'ит под sealing key CPU B.

Безопасность этого завязана на корректной remote attestation с обеих сторон. Это не простая "scp файл" операция.

### В: А microcode updates — ломают ли они sealing?

Иногда. Когда Intel выпускает митигацию для новой side-channel атаки, CPUSVN может продвинуться. Если ваша sealing policy требует строгого CPUSVN equality, новый энклейв не может распечатать данные, которые были sealed до bump'а. Если ваша sealing policy позволяет rollback к более низкому CPUSVN (флаг `SGX_KEYPOLICY_NOISVPRODID` и связанные), новый энклейв всё ещё может прочитать старые данные.

Есть trade-off: rollback tolerance означает, что sealed blob с vulnerable CPUSVN всё ещё может быть прочитан patched энклейвом, что может быть ровно тем, что вы *не* хотите, если уязвимость утекла ваш sealing key. На практике для долгоживущего операционного state (как vault), rollback tolerance обычно — правильный выбор, потому что альтернатива — терять state на каждом microcode update'е — операционно катастрофична.

## Часть 7 — Наша конкретная архитектура

### В: Как это реально раскладывается в perp DEX?

Из `project_fork_and_deploy.md` и текущей кодовой базы:

- **Enclave (C/C++) — `77ph/xrpl-perp-dex-enclave`**
  Держит FROST key shares, подписывает XRPL транзакции, управляет per-user margin ledger (по плану в `plans/jiggly-mapping-starfish.md`), поддерживает vault state, запускает логику ликвидации и funding. Sealed state для shares + vault + margin.

- **Orchestrator (Rust) — `LemonTreeTechnologies/xrpl-perp-dex`**
  Работает на хосте вне энклейва. Владеет matching engine, user sessions, WebSocket + REST API, XRPL client (мониторит ledger close события, submit'ит подписанные tx, возвращённые из энклейва, relay'ит результаты обратно через ECALL). Не держит key material.

- **Host OS (Ubuntu/Debian на Hetzner + Azure Confidential Computing)**
  Предоставляет SGX driver, AESM daemon, DCAP инфраструктуру. Untrusted с точки зрения энклейва.

### В: Где граница между "внутри энклейва" и "вне энклейва" для vault'а?

- **Внутри энклейва:** баланс vault'а, per-user позиции, signing keys, логика, решающая, разрешён ли вывод, логика, вычисляющая funding payments, логика, решающая, можно ли ликвидировать позицию, fixed-point math.
- **Вне энклейва:** CLOB orderbook, matching engine, price feeds, XRPL client, persistence layer (host держит ciphertext файлы), REST/WebSocket сервер, метрики/логи.

Orchestrator — "нервная система": воспринимает внешний мир и driv'ит энклейв через ECALL'ы. Энклейв — "policy core": решает, что разрешено, подписывает то, что одобряет, и никогда не касается wire.

### В: Какие sealed blob'ы существуют и какова их sealing policy (target state)?

*Target state, имплементируется планом enclave ecalls; текущее состояние зависит от того, что было подключено.*

| Blob | Содержит | Target policy | Обоснование |
|---|---|---|---|
| `frost_share.sealed` | FROST key share этого узла | MRSIGNER + versioned | Должен пережить enclave upgrades; recovery невозможен |
| `vault_state.sealed` | Баланс vault + открытые позиции | MRSIGNER + versioned | Должен пережить upgrades; реконструируется из chain с усилиями |
| `margin_ledger.sealed` | Per-user margin accounts | MRSIGNER + versioned | Должен пережить upgrades; трудно реконструировать |
| `tx_dedup.sealed` | Processed tx hash table (deposit replay guard) | MRSIGNER + versioned | Должен пережить upgrades для сохранения replay-гарантии |
| `nonce_ctr.sealed` | Monotonic signing counter | MRENCLAVE | Свежий на каждом binary; старые counters нерелевантны, и держать их связанными со старым binary закрывает минорную replay-поверхность |

### В: Когда "migration ceremony" паттерн оправдан для этого проекта?

Три случая:

1. **Ротация signing key.** Если когда-нибудь ротируем signing key энклейва (т.е. ключ, которым подписан сам enclave binary, определяющий MRSIGNER) — ceremony обязательна. Никаких shortcut'ов.
2. **Schema change, пересекающая invariant.** Если новая версия энклейва хочет интерпретировать старые данные способом, требующим вычисления новых полей из chain history — строим явный upgrade path, не полагаемся на seal-and-reload round-trip.
3. **Смена самой sealing policy** (например, добавление AEAD KDF параметров, смена major версии SDK) — dual-mode transition enclave.

Для рутинных изменений — bug fixes, новые ecalls, минорные logic tweaks — нормальный MRSIGNER sealing + schema versioning путь достаточен, и ceremony не нужна.

## Часть 8 — Известные ограничения и операционные грабли

### В: Суммируй те, что реально кусали этот проект или близкого соседа.

1. **DCAP attestation работает только на Azure, не на Hetzner.** SGX hardware на Hetzner не экспонирует DCAP provisioning service. Remote attestation практически только на Azure для нашего deployment'а. Зафиксировано в `feedback_dcap_subprocess_pattern.md`. Workaround на Hetzner: только local attestation, без remote attestation эндпоинта.
2. **DCAP subprocess pattern.** Никогда не разделяйте DCAP `target_info` / `get_quote` flow через subprocess'ы, и никогда не линкуйте DCAP QL (quote library) в enclave host напрямую — это корраптит state. См. `feedback_dcap_subprocess_pattern.md`.
3. **Azure + DCAP два бага.** `AZDCAP_DEBUG_LOG_LEVEL` должен быть установлен, иначе stdout корраптится, и новым VM иногда нужен deallocate+start для триггера PCK provisioning. См. `feedback_dcap_azure_two_bugs.md`.
4. **EPC не бесконечен.** Старые SGX CPU ограничивают EPC ~128MB. Новые server CPU (Ice Lake Xeon и позже) поддерживают до 512GB, но платят в page eviction, если вы выходите за hardware EPC. Eviction прозрачен, но медленный — CPU re-шифрует страницы, когда они входят и выходят. Для нашего vault'а с несколькими тысячами позиций это не проблема, но для CLOB с миллионами resting ордеров это было бы.
5. **Целевая платформа — только server Xeon.** SGX депрекейтнут на потребительских Core CPU (11/12-е поколения) ещё в 2021 и не годится как dev-платформа. Server SGX на Xeon Scalable остаётся поддерживаемой платформой; публичная позиция Intel — "no plans to deprecate" (support-статья 000089326). Не пытайтесь запустить это на ноутбуке. Отдельно: legacy **EPID** attestation service у Intel был EOL'нут 2 апреля 2025 — для remote attestation теперь только DCAP, и наш дизайн это уже учитывает.
6. **Каждая SGX уязвимость bump'ает CPUSVN.** Если ваша sealing policy не допускает rollback — каждый microcode update превращается в мини-миграционное событие. Мы выбрали rollback tolerance для долгоживущего state именно по этой причине.
7. **Attack surface не ноль.** Side-channel атаки на SGX продолжают открываться. SGX существенно поднимает планку против ОС и физического атакующего, но это не магический "не взламываемый" ящик. Дизайн FROST 2-of-3 — наша страховка на случай компрометации одного энклейва.

### В: Как мы сейчас собираем и деплоим энклейв?

Build происходит на SGX-сервере в HW режиме (`make SGX_MODE=HW SGX_DEBUG=1`), потому что build toolchain ожидает доступ к реальному SGX-устройству для шага подписи. Deployment — `scp` + restart сегодня. Post-hackathon план в `deployment-procedure.md` формализует это в 2-of-3 подписанный, воспроизводимо-собранный, attested pipeline.

## Часть 9 — Портируемость на Intel TDX (взгляд вперёд)

**Эта секция существует потому, что вопрос обязательно будет задан — PM'ом, инвестором или security-ревьюером — и неправильный ответ на него опасен. Правильный ответ состоит из двух частей, которые всегда должны звучать вместе: "да, это возможно при серьёзной инженерной работе" И "нет, это нельзя сделать срезанием углов, и наивный порт кода разрушает все гарантии безопасности, которые у нас сейчас есть".**

### В: Может ли та же perp-DEX логика работать внутри Intel TDX вместо SGX?

Да — при условии выделенного инженерного проекта с правильным scope, временем и финансированием. Алгоритмическое ядро `Enclave.cpp` (FROST, MuSig2, ECDSA, маржинальный движок, `PerpState`, account pool) — это примерно 80-90% portable C/C++ кода, не зависящего от SGX-семантики. Эти рутины скомпилировались бы без изменений внутри TDX-гостя. В ряде аспектов TDX *проще* в работе, чем SGX: настоящие `pthreads`, настоящая libc, настоящая файловая система, настоящая сеть внутри trust domain, нет EPC eviction penalty, есть live TD migration между хостами (TDX 1.5+).

**Но это абсолютно не упражнение "перекомпилировать и отправить", и любая попытка относиться к этому как к такому — де-факто разрушит все гарантии безопасности, которые у нас сейчас есть.** Эта мысль non-negotiable, и остальная часть секции объясняет почему.

### В: Что потребует TDX-версия с *эквивалентными* гарантиями безопасности?

Это multi-month инженерный проект, существенно больший, чем уже сделанная SGX-работа. Минимальный scope:

1. **Минимальный, hardened, воспроизводимо собранный гостевой образ.** В SGX доверенная база — это ~5k строк `Enclave.cpp` плюс тщательно аудированная `sgx_tstdlib`. В TDX доверенная база — это *вся* гостевая VM: ядро + libc + всё, что работает внутри. Чтобы удержать TCB достаточно маленьким, чтобы его можно было аудировать, гостевой образ надо строить с нуля как минимальный дистрибутив Linux: монолитное ядро без loadable modules, нет `/bin/sh`, нет `sshd`, нет `systemd`, нет менеджера пакетов, нет userland'а кроме единственного статически слинкованного бинаря как PID 1, seccomp-bpf default-deny, нет listen-портов кроме единственного vsock-канала, и всё это собирается воспроизводимо (NixOS или Buildroot), чтобы measurement был детерминированным. **Построение этого образа и его аудит до того же стандарта, что и SGX-энклейв — это самая большая часть работы, это отдельный проект, а не побочная задача.** Если этот шаг пропустить или сделать небрежно, гостевой образ содержит обычный Linux userland с обычным attack surface, и **весь аргумент конфиденциальности разваливается** — вы криптографически защищаете машину, которую враждебный оператор тривиально рутует изнутри. Это ровно тот сценарий провала, который продают как "просто запусти Docker-образ внутри TDX".

2. **Замена sealing-слоя.** SGX предоставляет `sgx_seal_data` / `sgx_unseal_data` со встроенной MRENCLAVE/MRSIGNER-привязанной key derivation и хорошо понятной migration story (наша Часть 6). TDX имеет аналогичные примитивы (семейство `MR_SEAL`, либо remote-KMS-keyed-to-TD-quote паттерны), но это *другие* примитивы с другой семантикой вывода ключа, с другими measurement chains (MRTD + RTMR вместо MRENCLAVE) и с другой операционной моделью. **Вся наша Часть 6 про миграцию должна быть переписана с нуля для TDX** — не концептуально переизобретена, а переделана под новую measurement-иерархию. Если этого не сделать и использовать любую ad-hoc persistence (например, "просто зашифровать файл hardcoded ключом") — гарантия снова разрушается, потому что теперь state может быть forked, replayed или извлечён любым, кто однажды получил копию TD-образа.

3. **Замена ECALL/OCALL границы.** В TDX нет ecall'ов. Узкая, аудируемая attack surface, которую даёт нам `Enclave.edl`, должна быть реконструирована как явный RPC-протокол (например, Protobuf поверх vsock) с такой же строгой валидацией, fuzzing'ом и единой точкой входа. **Если эта граница не будет спроектирована и аудирована так же тщательно, как EDL сегодня**, она становится новым слабым звеном — слишком широкая gRPC-поверхность или небрежный парсер внутри TD так же опасны, как баг в энклейве, и имеют тот же blast radius.

4. **Замена attestation-pipeline.** TDX-квоты DCAP-верифицируемы (та же verification-инфраструктура, что и SGX, другой формат квоты), но клиентский верификатор, quote-retrieval endpoint и release-signing pipeline — всему этому нужны TDX-специфичные пути. Ceremony из `deployment-procedure.md` выживает структурно (FROST 2-of-3 подписывает release manifest, чей хеш — это MRTD виртуалки вместо MRENCLAVE энклейва), но каждому инструменту в цепочке нужен TDX backend.

5. **Переписанная threat model и audit pass.** Граница доверия сдвигается с "одного процесса" до "одной VM". Каждое предположение в нашей текущей threat model надо пересмотреть: взаимодействия с host-гипервизором, IOMMU protection, TD-to-TD isolation, экспозиция guest kernel, trust при live migration, семантика rollback microcode. Это **не** упражнение в документации — это настоящий security audit с настоящими затратами, и его надо заложить в бюджет.

6. **`TeeBackend` абстракция в коде.** Каждый вызов `sgx_seal_data`, `sgx_unseal_data`, `sgx_read_rand`, `sgx_sha*`, `sgx_create_report`, `sgx_thread_mutex` и т.д. надо обернуть за TEE-агностичный интерфейс с SGX и TDX реализациями. Это единственная часть порта, которая реально дешёва (~1-2 недели), но она load-bearing: без неё остальной порт — запутанный рерайт вместо чистой замены бэкенда. **Это же единственный кусок, который стоит сделать спекулятивно на SGX-стороне уже сегодня**, до всякого TDX-финансирования, потому что он недорог и делает любой будущий порт посильным.

Реалистичный график для пунктов 1-5, выполненных *серьёзно*, — порядка **4-5 месяцев полного внимания senior-инженера, или 2-3 месяцев для команды из двух**, плюс независимый security audit сверху. Это настоящий проект, а не спринт.

### В: Можем ли мы просто сегодня запустить код в TD без всего этого?

Физически — да. *Безопасно* — **нет, и это надо проговаривать однозначно**. TDX-гость, запущенный из стокового Debian или Ubuntu образа с положенным туда нашим бинарём, включённым SSH, обычной init-системой и ad-hoc file-based storage'ем, *технически* будет confidential VM — гипервизор не может прочитать его память. Но:

- Любая компрометация внутри гостя (через ssh, через непропатченный пакет, через привилегированный daemon, через сетевую поверхность самой гостевой OS) утекает ключ подписи ровно так же эффективно, как компрометация хоста, потому что **весь userland гостя находится внутри trust boundary, и никто из этого не аудирован до требуемого стандарта**.
- Persistence без правильного TDX sealing или KMS-attested дизайна означает, что state можно скопировать, replay'нуть или подменить между запусками TD.
- Attestation без узкой measurement-схемы означает, что у пользователя нет криптографического способа отличить "тот TD, которому я должен доверять" от "TD, запущенного с того же образа, но с backdoor'ом, добавленным в userland до загрузки".
- История side-channel'ов не улучшается, но объём кода, который может быть эксплуатирован через side-channel, вырастает на порядки.

**Это критическая мысль, которую надо донести любому стейкхолдеру, спрашивающему про TDX:** свойства безопасности TEE не приходят от одного железа — они приходят от железа *в сочетании с* дисциплинированной инженерной работой по удержанию trusted base маленьким, измеренным и аудируемым. SGX *заставляет* эту дисциплину, потому что enclave-модель настолько ограничена; TDX *позволяет* эту дисциплину, но не forced её, и failure mode — тихий: "работает, значит безопасно" — это ловушка. TDX-деплоймент, который пропустил пункты 1-5, выглядел бы как рабочая система и де-факто давал бы **меньше** безопасности, чем наш текущий SGX-деплоймент, а не больше.

### В: Тогда какая наша реальная позиция сегодня?

- **Мы на SGX и останемся на SGX**, пока и если не будет правильно профинансирован и заскоплен выделенный TDX migration проект.
- **Без TDX-финансирования TDX недостижим как продукт.** Не потому что это технически невозможно — возможно — а потому что сделать это наполовину хуже, чем не делать вообще, а сделать правильно — это 4-5 месяцев инженерных инвестиций со своим аудитом.
- **TDX-версия, если будет построена, должна давать эквивалентные гарантии безопасности текущей SGX-версии, а не "TDX-style" гарантии.** Та же threat model, та же attestation story, та же migration story, та же маленькая-и-аудируемая trusted base. Что-то меньше этого — не альтернатива SGX, а регрессия, одетая в новое железо.
- **Единственное, что стоит делать сегодня, независимо от TDX-финансирования, — это ввести `TeeBackend` абстракцию** в enclave-коде, чтобы любой будущий порт стал добавлением бэкенда, а не рерайтом. Это дёшево, не блокирует текущую работу, и это хорошая инженерная гигиена сама по себе.
- **Если инвестор специально хочет TDX**, правильный разговор — "да, вот заскопленный проект и бюджет", а не "конечно, переключим флаг". Заскопленный проект — это то, что описывает эта секция.

## Приложение A — Однострочный глоссарий

- **EPC** — Enclave Page Cache, физически зарезервированный регион RAM, где живут enclave pages.
- **MEE** — Memory Encryption Engine, CPU-блок, прозрачно шифрующий EPC память.
- **ECALL** — вход из untrusted host code в enclave code.
- **OCALL** — выход из enclave code в untrusted host code (для syscalls, network, file I/O).
- **MRENCLAVE** — SHA-256 measurement кода энклейва и initial data.
- **MRSIGNER** — SHA-256 хеш публичного ключа, которым подписан энклейв.
- **CPUSVN** — CPU Security Version Number; продвигается, когда Intel поставляет microcode митигации.
- **ISV_SVN** — Independent Software Vendor Security Version Number; app-level счётчик под вашим контролем.
- **EPID** — старая схема remote attestation на group signatures; требует Intel-run attestation service.
- **DCAP** — Datacenter Attestation Primitives; новая PKI-based remote attestation, использована offline.
- **AESM** — Application Enclave Services Manager, Intel-поставляемый daemon на хосте, брокер attestation-связанных операций.
- **FROST** — Flexible Round-Optimized Schnorr Threshold signatures; наша 2-of-3 схема.
- **DKG** — Distributed Key Generation; как FROST shares создаются, чтобы никто не держал полный ключ.
- **TDX** — Intel Trust Domain Extensions; VM-level TEE, защищающий целую гостевую VM вместо одного процесса. Дополняет SGX, а не заменяет.
- **TD** — Trust Domain; одна гостевая VM, работающая под TDX.
- **MRTD** — Measurement начального образа TD (kernel + initrd + firmware); TDX-аналог MRENCLAVE.
- **RTMR** — Runtime Measurement Registers; дополнительные TDX-измерения, расширяемые во время загрузки гостя.
